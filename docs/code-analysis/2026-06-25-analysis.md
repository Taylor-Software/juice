## Headline

Codebase is healthy post-UX-refresh epic: full suite green, 86.6% coverage, zero lint warnings
(10 infos, all trivial). The dominant risk is **scale, not bugs** ‚Äî two files (`models.dart`
3920 LOC, `journal_screen.dart` 2957 LOC) have grown past the point where a single developer
can reason about them safely. Two hot-path performance issues (per-entry `mentionedCharIds`
recompute, dungeon bounds in the paint loop) are addressable without API changes. Dead-code
surface is tiny: one unused seam (~30 LOC).

**Top 3:** Split `models.dart` ‚Üí module files. Cache `mentionedCharIds`. Move dungeon bounds
out of `paint`.

## Findings

### üüÝ High

- **`lib/engine/models.dart`** ~3920 LOC ‚Äî 43 model classes + 5 enums in one file. Single
  `import` drags the entire domain graph into every feature. Extract into sub-modules:
  `models/journal.dart`, `models/character.dart`, `models/thread.dart`, `models/session.dart`,
  `models/map.dart`; barrel-export from `models.dart` for compatibility. Invasive but
  mechanical ‚Äî no logic change.

- **`lib/features/journal_screen.dart`** ~2957 LOC ‚Äî God widget. `_EntryTile` and its
  per-`JournalKind` renderers (`_PayloadCard`, `_DiceLogRow`, scene-divider, sketch-thumb)
  each exceed 100 LOC and are independently testable. Extract into
  `lib/features/journal_entry_tile.dart`. The `_composerSection` is another 200+ LOC
  candidate for extraction.

- **`mentionedCharIds` per-entry filter** ‚Äî `journal_screen.dart` calls `mentionedCharIds(e)`
  inside the `entries.where(...)` filter on every rebuild. This is O(n) per entry √ó total
  characters per character, redone on every provider tick. Cache result per-entry in a
  `select`-narrowed provider or memo map. On long journals this is a stutter-risk.

- **Dungeon bounds in paint loop** ‚Äî `DungeonMapPane`'s `CustomPainter.paint` recomputes
  `_boundingBox()` over all rooms on every frame. Bounds change only when rooms are added/
  moved. Cache in `MapNotifier` or derive lazily in `MapState`, invalidate on mutation.

### üü° Medium

- **`SessionsNotifier`** ~278 LOC, 14 public methods (`create`, `rename`, `setMode`,
  `editSystems`, `exportActive*`, `importCampaign*`, ‚Ä¶) ‚Äî god notifier. I/O methods
  (`exportActiveFile`, `importCampaignData`) belong in `campaign_io.dart` with notifier
  calling them as pure functions. Reduces test surface and isolation cost.

- **Force-unwrap on verified data** ‚Äî `oracle.dart`: `_fateMap[key]![likelihood.key]!` and
  `map_screen.dart`: `data.monsterEnvFormula['$envRow']!`. Both are backed by build-script-
  verified assets so they cannot fail in practice, but a corrupt or mis-named asset file
  yields an unhandled runtime exception with no diagnostic. Prefer
  `?? (throw StateError('missing key $key ‚Äî rebuild oracle_data.json'))` for fast-fail
  with context.

- **`kSystemPrimers[key]!` repeated 11√ó** ‚Äî `system_primer.dart` force-lookups the map
  directly at 11 call sites. Extract a `_primerFor(String key)` helper with one throw-site.

- **`allTags()` O(n¬∑m) per build** ‚Äî `lib/engine/models.dart` `JournalEntry.allTags()`
  is called inside list-build loops (e.g. tag-filter dropdowns). For 500+ entries this
  flattens and deduplicates 500 lists per frame. Compute once in `JournalNotifier.build`
  and expose as a derived provider.

- **`askGm()` single-shot seam unused** ‚Äî `lib/engine/oracle_interpreter.dart` exports
  `buildAskGmPrompt`/`parseAskGmResponse` (~30 LOC total). CLAUDE.md confirms app-unused
  (multi-turn `gmChat` replaced it). Safe to remove; cleans the public API surface.

- **`CharacterNotifier` 20+ mutation methods** ‚Äî `addIronsworn`, `addDnd`, `addShadowdark`,
  ‚Ä¶ `addArgosa`, `addCairn`, etc. Each is 5‚Äì10 LOC of `_mutate` boilerplate. A
  `addPreMadeSheet(SheetType)` dispatch or a factory method on `Character` would halve the
  notifier size without behavior change.

- **Lint: deprecated `TextFormField.value`** ‚Äî `kal_arath_sheet.dart:86,102`:
  `value:` ‚Üí `initialValue:`. Deprecated after Flutter v3.33. One-line fix √ó2.

- **Lint: missing `const` in test constructors** ‚Äî `test/campaign_preview_pane_test.dart:13‚Äì38`
  (8 instances). Trivial; `dart fix --apply` resolves all.

### üü¢ Low

- **`enter_campaign.dart` 21.4% coverage** ‚Äî core navigation helper. The integration test
  `resume_flow_test.dart` covers the happy paths, but unit paths (zero-entries ‚Üí `landFor`,
  `enterCampaignWith` guard when `entries` already empty) are untested. Low risk of regression
  but low cost to add.

- **`interpreter.dart` 46.2% coverage** ‚Äî `InterpreterService` abstract + `PhaseNotifier`
  helpers. `FakeInterpreterService` covers runtime behavior; the remaining gap is error-state
  branches (`InterpreterPhase.error`, download-fail paths) that only the real Gemma service
  exercises. Accept or add fake-triggered error tests.

- **`map_snapshot.dart` 33.3% coverage** ‚Äî `captureBoundaryPng` is a native-only path
  (calls `RenderRepaintBoundary.toImage`); headless tests can't pump it. The helper is
  2 LOC; the gap is acceptable given the device-verify record.

- **PDF rasterizer (0% / 33.3%)** ‚Äî `pdf_rasterizer_io.dart` and `pdf_rasterizer.dart`
  are native-only; correct by design (stub on web). No action needed.

- **`help_nav.dart` 0%** ‚Äî UI-only navigation widget, tested indirectly via device runs.
  A smoke widget-test would close the gap cheaply.

- **`battle_pane.dart` / `resources_pane.dart` ~1‚Äì2%** ‚Äî Lonelog subsystem, low adoption
  in the active test suite. Low priority until Lonelog usage grows.

### üí° Ideas

- **Tighten lint to `very_good_analysis`** ‚Äî the project already passes with 0 warnings at
  the current level. `very_good_analysis` adds `prefer_final_fields`,
  `avoid_dynamic_calls`, `unawaited_futures` checks. Worth evaluating; would surface 20‚Äì40
  new infos, most auto-fixable.

- **`models.dart` split doubles as a dependency-injection win** ‚Äî today every test that
  touches a notifier imports the full 3920-LOC models file. Splitting would cut cold-parse
  time and make it easier to mock only the slice a test cares about.

- **`SubtabHost` routing by string ID** ‚Äî already done; no index remapping on tab insertion.
  This pattern should be documented in CLAUDE.md as the canonical tab-addition contract to
  prevent future index-based regressions.

- **Thread `progressMax` default 10** ‚Äî the current default hard-codes the Ironsworn
  progress-track feel. A future "system-aware default" (`progressMax: session.isIronsworn
  ? 10 : 6`) would let Shadowdark or D&D characters use a different scale without a
  model change.

## Top 3 next moves

1. **Split `models.dart`** into 5 sub-modules (`journal`, `character`, `thread`, `session`,
   `map`) with a barrel export. Largest single leverage: reduces parse overhead, enables
   targeted imports, makes each model independently testable. Mechanical rename ‚Äî no logic
   change ‚Äî but touches ~80 import sites.

2. **Cache `mentionedCharIds` + dungeon bounds** ‚Äî fix both hot-path regressions in one PR.
   `mentionedCharIds`: expose a `mentionedCharIdsProvider` that `select`s over entries once.
   Dungeon bounds: move `_boundingBox()` onto `MapState`, invalidate in `addRoom`/`moveRoom`.
   Small, high-impact, low-risk.

3. **Extract `journal_screen.dart` entry renderers** ‚Äî pull per-`JournalKind` tile widgets
   into `lib/features/journal_entry_tile.dart`. Unblocks independent testing of entry
   rendering without pumping the full journal scaffold. Medium effort, high readability gain.
